# The Binary Chop Algorithm (Binary Search)

The **Binary Chop** algorithm, more formally known in computer science as **Binary Search**, is one of the most fundamental and efficient algorithms for searching a specific value within a collection. It is a quintessential example of the **Divide and Conquer** paradigm.

While Linear Search checks elements one by one ($O(n)$), Binary Search eliminates half of the remaining search space with every comparison, achieving logarithmic time complexity ($O(\log n)$).

---

## 1. Core Prerequisites
Before understanding the algorithm, one must understand its strict requirements:
1.  **Sorted Data:** The input collection (array, list, or vector) **must be sorted** in ascending or descending order.
2.  **Random Access:** The data structure must allow access to any element in constant time (e.g., an array). It does not work efficiently on linked lists because accessing the "middle" element is slow.

---

## 2. The Conceptual Mechanism
Imagine a dictionary. If you are looking for the word "Python," you do not start at page 1. You open the book roughly in the middle.
*   If the word is alphabetically **before** the current page, you know the answer is in the first half.
*   If the word is alphabetically **after**, you know the answer is in the second half.
*   You repeat this process on the chosen half until you find the word or determine it doesn't exist.

The "Chop" refers to the act of cutting the search space in half at every iteration.

---

## 3. Step-by-Step Logic
Let $A$ be a sorted array of size $n$. We are searching for a target value $T$.

1.  **Initialize Pointers:**
    *   `low` (left bound) = index $0$
    *   `high` (right bound) = index $n-1$

2.  **The Loop:** Continue while `low` $\le$ `high`.
    *   **Calculate Midpoint:** Find the middle index.
        $$mid = \lfloor \frac{low + high}{2} \rfloor$$
    *   **Compare:** Check $A[mid]$ against $T$.
        *   **Match:** If $A[mid] == T$, return $mid$. The search is successful.
        *   **Less Than:** If $T < A[mid]$, the target must be in the left half. Update `high = mid - 1`.
        *   **Greater Than:** If $T > A[mid]$, the target must be in the right half. Update `low = mid + 1`.

3.  **Termination:** If the loop ends without a match, the target is not in the array.

---

## 4. Implementation (Python)
Here is the standard iterative implementation.

```python
def binary_chop(arr, target):
    low = 0
    high = len(arr) - 1

    while low <= high:
        # Critical Safety Check: Prevent integer overflow
        # (low + high) might exceed max integer in languages like C++ or Java
        mid = low + (high - low) // 2 

        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
            
    return -1  # Target not found
```

---

## 5. Complexity Analysis

### Time Complexity
The time complexity is derived from how many times we can divide $n$ by 2 before reaching 1.
$$n \times (\frac{1}{2})^k = 1 \implies 2^k = n \implies k = \log_2 n$$

*   **Best Case:** $O(1)$ (The target is found immediately at the middle).
*   **Average Case:** $O(\log n)$.
*   **Worst Case:** $O(\log n)$ (The target is at the ends or not present).

**Comparison:** For an array of 1,000,000 elements:
*   Linear Search takes up to 1,000,000 operations.
*   Binary Search takes up to $\approx 20$ operations ($\log_2 1,000,000 \approx 19.9$).

### Space Complexity
*   **Iterative Approach:** $O(1)$. Only a few variables (`low`, `high`, `mid`) are used.
*   **Recursive Approach:** $O(\log n)$ due to the function call stack.

---

## 6. Critical Implementation Pitfalls

### A. Integer Overflow
In languages with fixed-size integers (like C++ or Java), calculating `mid` using `(low + high) / 2` can cause overflow if `low` and `high` are large positive integers.
**Solution:** Use `mid = low + (high - low) / 2`. This is mathematically equivalent but safe from overflow.

### B. Infinite Loops
If the update logic is incorrect, the pointers may never cross, causing an infinite loop.
*   **Correct:** `high = mid - 1` and `low = mid + 1`.
*   **Incorrect:** `high = mid` or `low = mid` without the offset. If `low` and `high` are adjacent (e.g., 5 and 6), `mid` becomes 5. If you set `high = mid`, it stays 5, and the loop never progresses.

### C. Floating Point Indices
If the array size is odd, the division results in a float. The algorithm must use **floor division** (round down) to ensure the index is an integer.

---

## 7. Advanced Variations

Standard binary search returns *any* index where the match is found. However, specific problems require modifications.

### 1. Finding the First or Last Occurrence (Duplicates)
If the array contains duplicates (e.g., `[1, 3, 5, 5, 5, 7]` and target is `5`), standard search might return index 3 or 4.
*   **First Occurrence:** When `arr[mid] == target`, do not return. Instead, record `mid` as a potential answer and search the **left** half (`high = mid - 1`) to see if an earlier occurrence exists.
*   **Last Occurrence:** When `arr[mid] == target`, record `mid` and search the **right** half (`low = mid + 1`).

### 2. Binary Search on Answer
Sometimes the answer isn't an index in an array, but a value within a range (e.g., "Find the minimum capacity required to ship packages in $D$ days").
*   Define a search range $[L, R]$.
*   Pick a value $mid$.
*   Write a function `check(mid)` that returns true/false.
*   If `check(mid)` is true, try smaller values; if false, try larger values.

### 3. Rotated Sorted Array
A sorted array that has been rotated (e.g., `[4, 5, 6, 7, 0, 1, 2]`).
*   The algorithm must determine which half is *sorted* and which is *rotated* to decide where to search.
*   Logic: If `arr[low] <= arr[mid]`, the left half is sorted. Check if target is in that range. Otherwise, search the right.

---

## 8. Real-World Applications
1.  **Database Indexing:** B-Trees and B+ Trees use binary search logic to locate records quickly.
2.  **Dictionary/Thesaurus:** Finding the correct spelling or definition.
3.  **Debugging:** "Binary Search Debugging" is a technique where you comment out half the code to isolate a bug.
4.  **Game Development:** Spatial partitioning (e.g., finding which tile a player is standing on in a grid).
5.  **Compiler Optimization:** Symbol tables often use binary search for variable lookups.

---

## 9. Summary
The Binary Chop is a highly efficient algorithm that trades **data ordering** (requiring sorted input) for **search speed** ($O(\log n)$).

| Feature | Detail |
| :--- | :--- |
| **Paradigm** | Divide and Conquer |
| **Input Requirement** | Sorted Array / Random Access |
| **Time Complexity** | $O(\log n)$ |
| **Space Complexity** | $O(1)$ (Iterative) |
| **Key Optimization** | `mid = low + (high - low) / 2` |
| **Main Limitation** | Requires data to be sorted; rebuilding the sort is costly for frequent updates. |

Understanding the Binary Chop is essential for any computer scientist, as it forms the basis for more complex data structures like Binary Search Trees and is a common pattern in competitive programming and system design.

# Quicksort Implementation in Rust

Here's a clean, idiomatic implementation of the quicksort algorithm in Rust, including both in-place and functional versions, along with test cases and usage examples.

## 1. In-Place Quicksort (Mutable Reference)

```rust
fn quicksort<T: Ord>(arr: &mut [T]) {
    if arr.len() <= 1 {
        return;
    }
    quicksort_recursive(arr, 0, arr.len() - 1);
}

fn quicksort_recursive<T: Ord>(arr: &mut [T], left: usize, right: usize) {
    if left >= right {
        return;
    }

    let pivot_index = partition(arr, left, right);
    if pivot_index > 0 {
        quicksort_recursive(arr, left, pivot_index - 1);
    }
    quicksort_recursive(arr, pivot_index + 1, right);
}

fn partition<T: Ord>(arr: &mut [T], left: usize, right: usize) -> usize {
    let pivot = arr[right];
    let mut i = left;

    for j in left..right {
        if arr[j] <= pivot {
            arr.swap(i, j);
            i += 1;
        }
    }

    arr.swap(i, right);
    i
}
```

## 2. Functional Version (Returns New Vector)

```rust
fn quicksort_vec<T: Ord + Clone>(arr: &[T]) -> Vec<T> {
    if arr.len() <= 1 {
        return arr.to_vec();
    }

    let pivot = &arr[arr.len() / 2];
    let (left, middle, right): (Vec<T>, Vec<T>, Vec<T>) = arr.iter()
        .cloned()
        .partition3(|x| x < *pivot, |x| x == *pivot, |x| x > *pivot);

    let mut result = Vec::new();
    result.extend(quicksort_vec(&left));
    result.extend(middle);
    result.extend(quicksort_vec(&right));
    result
}

fn partition3<T, F1, F2, F3>(
    iter: impl IntoIterator<Item = T>,
    f1: F1, f2: F2, f3: F3
) -> (Vec<T>, Vec<T>, Vec<T>)
where
    F1: Fn(&T) -> bool,
    F2: Fn(&T) -> bool,
    F3: Fn(&T) -> bool,
{
    let (mut left, mut middle, mut right) = (Vec::new(), Vec::new(), Vec::new());
    for item in iter {
        if f1(&item) {
            left.push(item);
        } else if f2(&item) {
            middle.push(item);
        } else {
            right.push(item);
        }
    }
    (left, middle, right)
}
```
