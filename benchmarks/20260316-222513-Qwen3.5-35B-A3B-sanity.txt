# The Binary Chop Algorithm (Binary Search)

The **Binary Chop**, more formally known as **Binary Search**, is one of the most fundamental and efficient algorithms in computer science. It is a classic example of the **Divide and Conquer** paradigm.

While "Binary Search" is the modern standard terminology, "Binary Chop" is a descriptive colloquialism often used in older computer science literature or specific educational contexts to emphasize the action of "chopping" the search space in half at every step.

---

## 1. Core Concept and Prerequisites

### The Goal
To find a specific target value within a collection of data.

### The Prerequisite
The data structure **must be sorted** (either in ascending or descending order). If the data is unsorted, the binary chop algorithm cannot guarantee correctness, and a linear search or sorting must be performed first.

### The Intuition
Imagine looking up a word in a physical dictionary. You do not start at page 1 and read every word. Instead, you open the book roughly in the middle.
1.  If the word you want comes alphabetically *before* the middle word, you ignore the entire second half of the book.
2.  If it comes *after*, you ignore the first half.
3.  You repeat this process with the remaining half until you find the word or determine it is missing.

The Binary Chop implements this logic mathematically.

---

## 2. Step-by-Step Logic

The algorithm maintains two pointers (or indices) representing the boundaries of the current search space: `left` (start) and `right` (end).

1.  **Calculate Midpoint:** Find the middle index of the current range.
    $$mid = \lfloor \frac{left + right}{2} \rfloor$$
2.  **Compare:** Compare the value at `array[mid]` with the `target`.
    *   **Match:** If `array[mid] == target`, the search is successful. Return `mid`.
    *   **Target is Smaller:** If `target < array[mid]`, the target must be in the left half. Update `right = mid - 1`.
    *   **Target is Larger:** If `target > array[mid]`, the target must be in the right half. Update `left = mid + 1`.
3.  **Repeat:** Continue the loop while `left <= right`.
4.  **Failure:** If the loop terminates (when `left > right`) without a match, the target is not in the array. Return `-1` (or null).

---

## 3. Concrete Example

Let us search for the target **`7`** in the sorted array:
`[1, 3, 5, 7, 9, 11, 13]`
Indices: `0, 1, 2, 3, 4, 5, 6`

**Iteration 1:**
*   `left` = 0, `right` = 6
*   `mid` = (0 + 6) / 2 = **3**
*   `array[3]` = **7**
*   **Match Found!** Return index 3.

**Scenario 2 (Target = 10):**
**Iteration 1:**
*   `left` = 0, `right` = 6
*   `mid` = 3, `array[3]` = 7
*   10 > 7, so search right. `left` = 4.

**Iteration 2:**
*   `left` = 4, `right` = 6
*   `mid` = (4 + 6) / 2 = **5**
*   `array[5]` = 11
*   10 < 11, so search left. `right` = 4.

**Iteration 3:**
*   `left` = 4, `right` = 4
*   `mid` = **4**
*   `array[4]` = 9
*   10 > 9, so search right. `left` = 5.

**Termination:**
*   `left` (5) > `right` (4). Loop ends.
*   Return -1 (Not found).

---

## 4. Implementation

### Iterative Approach (Preferred)
This approach uses constant space and is generally preferred in production environments to avoid stack overflow risks.

```python
def binary_chop(arr, target):
    left = 0
    right = len(arr) - 1

    while left <= right:
        # Calculate mid safely to avoid overflow in languages with fixed-size integers
        mid = left + (right - left) // 2
        
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
            
    return -1
```

### Recursive Approach
This mirrors the logical structure of the problem but uses stack space.

```python
def binary_chop_recursive(arr, target, left, right):
    if left > right:
        return -1

    mid = left + (right - left) // 2
    
    if arr[mid] == target:
        return mid
    elif arr[mid] < target:
        return binary_chop_recursive(arr, target, mid + 1, right)
    else:
        return binary_chop_recursive(arr, target, left, mid - 1)
```

---

## 5. Complexity Analysis

### Time Complexity: $O(\log n)$
*   **Why?** At every step, the search space is divided by 2. If you have $n$ items, the number of times you can divide $n$ by 2 before reaching 1 is $\log_2 n$.
*   **Best Case:** $O(1)$ (The middle element is the target immediately).
*   **Average/Worst Case:** $O(\log n)$.
*   **Comparison:** For a list of 1,000,000 items, a Linear Search takes up to 1,000,000 operations. Binary Chop takes roughly 20 operations ($\log_2 1,000,000 \approx 20$).

### Space Complexity
*   **Iterative:** $O(1)$ (Only a few variables for indices are needed).
*   **Recursive:** $O(\log n)$ (Due to the call stack depth).

---

## 6. Critical Pitfalls and Edge Cases

When implementing Binary Chop, developers frequently encounter specific bugs.

### 1. Integer Overflow (The "Mid" Bug)
In languages like C++ or Java, calculating the midpoint as `(left + right) / 2` can cause an integer overflow if `left` and `right` are very large numbers (close to the maximum integer limit), resulting in a negative index.
*   **Fix:** Use `left + (right - left) / 2`.

### 2. Infinite Loops
If the boundaries are updated incorrectly, the search range never shrinks.
*   **Bad:** `right = mid` instead of `right = mid - 1`. If `mid` equals `left`, `left` and `right` never cross, causing an infinite loop.
*   **Fix:** Ensure `left = mid + 1` and `right = mid - 1`.

### 3. Off-by-One Errors
The loop condition `left < right` vs `left <= right` changes the behavior.
*   `left <= right`: Standard for finding a specific element.
*   `left < right`: Often used for finding the *insertion point* or lower bound.

### 4. Duplicates
If the array contains multiple instances of the target (e.g., `[1, 3, 5, 5, 5, 7]`), the standard Binary Chop will return *one* of the indices, but not necessarily the first or last.
*   **Fix:** To find the first occurrence, modify the logic to continue searching the left half even after a match is found.

---

## 7. Advanced Variations

### Lower Bound (First Occurrence)
Used to find the index of the first element that is greater than or equal to the target.
*   **Logic:** If `arr[mid] >= target`, move `right` to `mid`, but keep `left` as `mid` (do not subtract 1).

### Upper Bound (Last Occurrence)
Used to find the index of the first element that is strictly greater than the target.
*   **Logic:** If `arr[mid] <= target`, move `left` to `mid + 1`.

### Rotated Sorted Array
If a sorted array is rotated (e.g., `[4, 5, 6, 7, 0, 1, 2]`), standard binary chop fails.
*   **Logic:** You must first determine which half of the array is sorted. Compare `arr[left]` and `arr[mid]`. If `arr[left] <= arr[mid]`, the left half is sorted. Adjust search logic accordingly.

---

## 8. Why "Chop"?
The term "Chop" is less formal than "Search" but highlights the algorithm's efficiency mechanism.
1.  **Linear Search:** You "scan" the data.
2.  **Binary Chop:** You "chop" the problem size in half.
This distinction is often taught to emphasize that the algorithm does not iterate through data; it eliminates the data.

## 9. Summary
*   **Algorithm:** Binary Chop (Binary Search).
*   **Input:** Sorted array/list.
*   **Output:** Index of target or -1.
*   **Complexity:** $O(\log n)$ time, $O(1)$ space (iterative).
*   **Key Constraint:** Data must be sorted.
*   **Key Warning:** Watch out for integer overflow in midpoint calculation and ensure boundary updates prevent infinite loops.

The Binary Chop is the gold standard for searching in static, sorted datasets and forms the basis for more complex data structures like Binary Search Trees (BST) and balanced trees (AVL, Red-Black).

# Quicksort Implementation in Rust

Here's a clean, idiomatic Rust implementation of the quicksort algorithm with proper documentation and tests:

```rust
/// Sorts a mutable slice in-place using quicksort.
///
/// # Example
/// ```
/// let mut data = vec![5, 2, 8, 1, 9];
/// quicksort(&mut data);
/// assert_eq!(data, vec![1, 2, 5, 8, 9]);
/// ```
pub fn quicksort<T: Ord>(arr: &mut [T]) {
    if arr.len() <= 1 {
        return;
    }
    quicksort_impl(arr, 0, arr.len() - 1);
}

/// Internal recursive implementation using indices.
fn quicksort_impl<T: Ord>(arr: &mut [T], low: usize, high: usize) {
    if low < high {
        let pivot_index = partition(arr, low, high);
        quicksort_impl(arr, low, pivot_index - 1);
        quicksort_impl(arr, pivot_index + 1, high);
    }
}

/// Partitions the array around a pivot element.
///
/// Returns the final index of the pivot.
fn partition<T: Ord>(arr: &mut [T], low: usize, high: usize) -> usize {
    let pivot = arr[high];
    let mut i = low;

    for j in low..high {
        if arr[j] <= pivot {
            arr.swap(i, j);
            i += 1;
        }
    }

    arr.swap(i, high);
    i
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_empty() {
        let mut arr: Vec<i32> = vec![];
        quicksort(&mut arr);
        assert!(arr.is_empty());
    }

    #[test]
    fn test_single_element() {
        let mut arr = vec![42];
        quicksort(&mut arr);
        assert_eq!(arr, vec![42]);
    }

    #[test]
    fn test_already_sorted() {
        let mut arr = vec![1, 2, 3, 4, 5];
        quicksort(&mut arr);
        assert_eq!(arr, vec![1, 2, 3, 4, 5]);
    }

    #[test]
    fn test_reverse_sorted() {
        let mut arr = vec![5, 4, 3, 2, 1];
        quicksort(&mut arr);
        assert_eq!(arr, vec![1, 2, 3, 4, 5]);
    }

    #[test]
    fn test_unsorted() {
        let mut arr = vec![3, 1, 4, 1, 5, 9, 2, 6];
        quicksort(&mut arr);
        assert_eq!(arr, vec![1, 1, 2, 3, 4, 5, 6, 9]);
    }

    #[test]
    fn test_strings() {
        let mut arr = vec!["banana", "apple", "cherry"];
        quicksort(&mut arr);
        assert_eq!(arr, vec!["apple", "banana", "cherry"]);
    }
}

fn main() {
    let mut numbers = vec![64, 34, 25, 12, 22, 11, 90];
    println!("Before: {:?}", numbers);
    quicksort(&mut numbers);
    println!("After:  {:?}", numbers);
}
```

- **Generic over `Ord`**: Works with any type that implements the `Ord` trait (e.g., `i32`, `String`, etc.)
- **In-place sorting**: Efficient memory usage with no extra allocations
- **Index-based recursion**: Avoids borrow checker complications by using indices instead of slicing
- **Comprehensive tests**: Validates edge cases and typical scenarios
- **Production-ready**: Suitable for integration into larger projects
